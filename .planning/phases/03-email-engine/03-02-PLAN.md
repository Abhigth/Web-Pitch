## Objective
Implement Nodemailer processing inside an independent, rate-limited BullMQ queue so emails dispatch without spam-flagging or UI disruption.

<task type="auto">
  <name>Task 2: Build the Nodemailer Worker</name>
  <files>worker.ts</files>
  <read_first>worker.ts, prisma/schema.prisma</read_first>
  <action>
    - In `worker.ts`, add a new `Worker` instance bridging `'email-campaigns'`.
    - In the worker configuration arguments, set the limiter to mimic a human pace: `{ connection, limiter: { max: 2, duration: 60000 } }` (2 emails per minute).
    - The worker's job data expectation: `{ leadId: string, templateText: string }`.
    - Inside the processor logic: Fetch the `Lead` by ID, then fetch the `SmtpSettings` for `lead.campaign.userId`. If missing, throw an error.
    - Initialize `nodemailer.createTransport({ host, port, secure: port === 465, auth: { user, pass } })`.
    - Call `await transporter.sendMail({ from: user, to: lead.website || lead.email || "test@example.com", subject: "Web Design Inquiry", text: job.data.templateText.replace('{name}', lead.name) })`.
    - On success, `prisma.lead.update({ where: { id: leadId }, data: { status: 'CONTACTED' } })`.
  </action>
  <acceptance_criteria>
    - `worker.ts` adds a second Worker for `'email-campaigns'`
    - The BullMQ worker limiter is set (e.g., `max: 2, duration: 60000`)
    - `nodemailer.createTransport` and `sendMail` use the user credentials
    - `prisma.lead.update` changes the status to `CONTACTED`
  </acceptance_criteria>
</task>
